# ZOM-BOY+ 规则/策略型 AI 调研与设计方案

日期：2026-05-15
范围：纯调研 + 出方案，**不改任何代码**。目标是为已完成的双人热座对战游戏 ZOM-BOY+ 设计一个传统博弈/启发式 AI（非大模型），让人能单人对战。

参考代码：`src/game/types.ts`、`src/game/rules.ts`、`src/game/state.ts`、`src/game/mapgen.ts`、`src/main.ts`、`src/ui/board.ts`、`src/ui/hud.ts`、`src/ui/cardModal.ts`，规则 spec：`docs/superpowers/specs/2026-05-15-zomboy-design.md`。

## 0. 现状速读（AI 设计必须贴着的事实）

游戏状态完全集中在一个可变对象 `State`（`types.ts:49`）。所有玩法都通过**唯一入口** `clickCell(state, r, c)`（`state.ts:244`）驱动，UI（`main.ts:onCellClick`）只是反复调用它并 `render()`。这对接 AI 极其友好——AI 不需要任何并行的"动作系统"，只要能生成**一串等价于玩家点击的 (r,c) 序列**，复用 `clickCell` 即可，规则一致性 0 风险。

关键纯函数（`rules.ts`）已经齐备，AI 可直接复用：

- `legalMoves(state, piece)` —— 僵尸正交 4 向 / 幸存者 8 向的单步空格。
- `legalJumps(state, piece)` —— 幸存者跳杀（返回 `{destR,destC,killR,killC}`）。
- `emptyCells(state)` —— 所有可放置空格（召唤 / 传送 / 鬼魂落点）。
- `runInfection(state)` —— 回合末感染结算（**有副作用，会改 state**）。
- `checkWinner(state)` —— 4 杀判定。
- `pieceAt / pieceById / otherSide`。

回合机重要细节（决定 AI 怎么"点"）：

- **僵尸回合的动作类型是从点击推断的，不是显式选择**（`types.ts:21-25`、`state.ts:handleZombieClick`）：
  - 点空格 + 库存>0 → 召唤 1 僵尸，**回合立即结束**。
  - 点空格 + 库存=0 → 暂存传送目标 `pendingTeleport`，**再点一只己方僵尸**完成传送，回合结束。
  - 点己方僵尸 → 进入 move 模式 `movesLeft=2`，之后每点一个合法目标格走 1 步，2 步用完回合结束（可同子两步或两子各一步）。
- **幸存者回合**：先点己方幸存者（设 `selectedPieceId`），再点目标格；`handleSurvivorClick` 自动判定跳杀 vs 普通移动。踩到未消耗的房子 → `state.deck.shift()` 抽卡，进入 `pendingCard` 多阶段交互（鬼魂 / 靴子 / 咖啡），由 `resolvePendingCardClick` 推进。
- 每个动作末尾走 `runEndOfTurn`：感染结算 → 胜负 → 切边 → 重置 zombieTurn。
- **随机性来源**：① 牌堆 `deck` 开局 `shuffle`（`state.ts:139`，固定 5 张：ghost×2, shoes×2, coffee×1）；② 地图 `generateMap` 随机（开局即定，对局内不变）；③ flavor 文案随机（与玩法无关，可忽略）。**对局内唯一的隐藏随机是牌堆顺序**，且只在幸存者踩房子时才兑现。

牌堆是**确定性已知集合的随机排列**：剩余牌的多重集合完全可由 `state.deck` 读出（AI 若是某一方，规则上 deck 内容对它"可见"——这是实现选择，不是信息隐藏）。这一点极大简化了随机处理。

胜负目标**非对称**：幸存者要 4 次跳杀，僵尸要 4 次感染（正交相邻 ≥2 僵尸）。库存 9 + 在场决定僵尸"产能"。

## 1. 状态空间与分支因子估算

### 1.1 僵尸回合分支因子

僵尸回合二选一，合法动作集合 = 召唤集 ∪ 移动集（库存=0 时召唤变传送）。

- **召唤**：落点 = `emptyCells(state)`。8×8=64 格，扣掉 ~石头（stoneRatio 0.18，约 9 块）、4 房子、4 start、当前在场棋子。开局空格 ≈ 64 − 9 − 4 − 4 − 4(幸存) − 0(僵尸) ≈ **40 多个候选**；中盘随僵尸增多降到 **20~35**。
- **移动 2 步**：在场僵尸数 Z 个，每个僵尸正交合法步均值约 2~3（受边界/石头/棋子挤压）。两点行动可分配给同一子或两子，组合数量级 ≈ `(Z·avgMoves)²` 量级但实际唯一终局态更少，落到 **数十到一百多**。
- **传送（库存=0）**：选 1 僵尸 × 落点 ≈ `Z × 空格数`，可达 **数百**（这是分支最大的分支，但出现在后期）。

**僵尸回合有效分支因子量级：召唤主导期约 30~50；移动期约 30~120；传送期可达 200~500。** 但其中绝大多数召唤/传送落点是"垃圾着"——离任何幸存者都很远，对包夹毫无贡献。**用规则裁剪后真正值得考虑的着法通常 < 15。**

### 1.2 幸存者回合分支因子

- 在场幸存者 S 个（开局 4，被感染递减）。每子普通移动 8 向均值 4~6 个合法格，跳杀 0~3 个（多数回合 0）。
- 总动作 ≈ `S × (avgMoves + avgJumps)` ≈ 4 ×（5 + 0~1）≈ **约 16~28**，跳杀机会出现时局部更尖锐。
- 房子抽卡引入分叉，但**牌堆已知**：可按"抽到 ghost / shoes / coffee"三个确定分支展开（含概率），不是真未知。

**幸存者回合有效分支因子量级：约 15~30，裁剪后核心着法 < 10。**

### 1.3 博弈树深度

- 一"回合对"= 僵尸走 + 幸存者走 = 2 ply。
- 综合分支 b ≈ 30~80（不裁剪）/ 8~15（规则裁剪后）。
- 全宽 minimax：b^d。不裁剪时 4 ply ≈ 30^4 ≈ 81 万 ~ 80^4 ≈ 4000 万，浏览器单帧吃不下；**裁剪后 b≈10，6 ply ≈ 10^6**，配 alpha-beta（有效 ≈ b^(d/2)）实际 ≈ 10^3 量级，**完全可在 < 50ms 内搜完**。
- 对局总长度：达成 4 杀通常 15~40 回合对（30~80 ply），不算长，但状态有环（来回走子），**不能做完整 endgame 搜索**，只能定深 + 评估函数截断。

**结论**：分支因子本身偏大但高度"可裁剪"（大量召唤/传送落点是无意义着）。这指向的算法路线是：**规则裁剪候选 → 浅层定深搜索（α-β / expectimax）→ 评估函数截断**。纯全宽搜索不可行，纯 MCTS 单局模拟成本与浏览器即时性矛盾，规则+浅搜的混合是甜点区。

## 2. 候选算法逐个评估

评估维度：可行性、强度上限、实现成本、浏览器实时性（必须 < 100ms 出招，理想 < 50ms，否则卡顿）。

### 2.1 纯启发式规则 AI（手写策略）

**做法**：不搜索，按一组优先级规则直接选当前最优着。

- 僵尸方优先级：① 本回合能直接达成感染（让某幸存者正交僵尸数从 1→2）的着 → 必走；② 把一个幸存者的"被包夹进度"推进（1 个正交僵尸 → 占第 2 个相邻正交空格）；③ 堵幸存者的跳杀落点 / 占据幸存者会落的关键格；④ 召唤位置贴近幸存者群但避免立刻被跳杀（不要送进"幸存者相邻且其背后空"的格）；⑤ 否则召唤在最能扩大控场的中心格。
- 幸存者方优先级：① 任何安全跳杀（跳完落点不会立刻被 ≥2 僵尸正交包夹）→ 取分；② 远离正交僵尸数 ≥1 的危险（避免下回合被感染）；③ 向最近"可形成跳杀"的僵尸机动，但保持落点不贴边（贴边减少跳杀方向与逃生方向）；④ 顺路踩未消耗房子吃卡（牌总体利好幸存者）。

**评估**：

- 可行性：高。所有判断都能用现成 `legalMoves/legalJumps/runInfection`（注意 `runInfection` 有副作用，AI 内部要在 `structuredClone` 出来的状态上试算）。
- 强度上限：中。能打出"会包夹/会跳杀/不送死"的像样对手，足够新手到中等玩家有挑战。但缺乏前瞻，会被有计划的人类用 2~3 步组合诱杀（如假装露破绽引僵尸聚集后连跳）。
- 实现成本：**最低**。约 1 个文件、几个评分函数、~200~300 行。0.5~1 人天。
- 实时性：**极佳**，无搜索，<1ms。

**定位**：MVP 首选 / 简单与中等难度档的基础，也是后续搜索的"候选生成器"和"叶子兜底"。

### 2.2 Minimax + Alpha-Beta + 评估函数

**做法**：定深搜索，叶子用评估函数（第 3 节）打分，α-β 剪枝。

**随机性如何处理**——这是本游戏的关键技术点：

- 对局内随机只有"踩房子抽到哪张牌"。把抽卡建模为 **chance node**，得到 **expectiminimax**（max 层 / min 层 / chance 层交替）。chance 节点的子节点概率 = 剩余牌堆多重集合里各 `CardKind` 的占比（从 `state.deck` 直接算，**信息已知，无需 determinization**）。
- 简化：多数着法**不踩房子**，根本不触发 chance 节点；只有"幸存者落到未消耗 house"的着法才展开 chance 层。所以 expectiminimax 的 chance 层很稀疏，开销可控。
- 进一步简化（难度低档可用）：忽略卡牌效果，把房子当普通格 + 一个"踩房子价值常量"加进评估，退化回纯 α-β。这损失精度但实现快、够用。

**评估**：

- 可行性：高，但需正确处理**非对称胜利条件**（评估函数对两方语义不同，见第 3 节）、**僵尸回合一次走 2 步是一个复合动作**（搜索的"一步"= 一整个僵尸回合的结果态，要在动作生成器里把"召唤 / (子a走+子b走) / (子a走两步) / 传送"枚举成原子转移）。
- 强度上限：高。3~5 ply 配好评估函数能算清"我包夹你你跳我我补位"的短组合，明显强于纯规则。
- 实现成本：中。需要：纯函数化的"应用一个完整回合动作 → 新状态"（不能复用带 UI 副作用的 `clickCell` 做内部搜索，要写 `applyAction(state,action): State`，但可在新状态上调 `runInfection` 复用规则）；动作枚举器；α-β 框架；评估函数。约 2~3 人天。
- 实时性：裁剪后 b≈10、深度 4~6，配 α-β + 着法排序（先试跳杀/感染着），单步 < 50ms 可达。**必须做候选裁剪**，否则传送期 b≈数百会炸。

**定位**：困难档主力。建议在规则 AI 跑通后增量加。

### 2.3 蒙特卡洛树搜索 MCTS

**做法**：UCT 选择 + 随机/启发 rollout，迭代预算内出招。

**评估**：

- 可行性：可行。对高分支、随机性鲁棒（牌堆随机用 determinization 处理）。
- 强度上限：高，尤其 rollout 用规则 AI（"重 playout"）时。但本游戏单局长度 30~80 ply、且存在来回走子的环，**随机 rollout 收敛慢、噪声大**；要 rollout policy 很强才好用，否则不如浅 α-β。
- 实现成本：中高。需要 `applyAction` 纯函数（同 2.2）+ 树结构 + UCT + rollout policy + 时间预算控制。约 3~4 人天。
- 实时性：可控（按时间预算 anytime），但浏览器单线程，给 30~50ms 只能跑有限模拟（每局 30~80 步、每步要枚举裁剪着法），单局模拟成本不低。强度/耗时比在本游戏不如"规则裁剪 + 浅 α-β"。

**定位**：非首选。仅当后续要追求很高棋力、且 α-β 评估函数难调时再考虑。

### 2.4 混合：规则裁剪 + 浅层搜索（推荐的最终形态）

**做法**：

1. 用 2.1 的规则启发式生成**候选动作 Top-K**（K≈6~10），把上百个垃圾召唤/传送落点直接砍掉。
2. 在候选上跑 2.2 的 α-β / expectiminimax，深度 3~5。
3. 叶子用评估函数；搜索超时或叶子用规则 AI 的"一步评分"兜底。
4. 难度档通过 K、深度、ε-随机调。

**评估**：可行性高、强度高、实时性好（裁剪后 b 小，搜索浅但有效）、实现成本= 2.1 + 2.2 之和但 2.1 本就是 2.2 的候选生成器，复用度高。**这是本游戏的算法甜点。**

### 对比小结

| 算法 | 可行性 | 强度上限 | 实现成本 | 浏览器实时性 | 适用档 |
|---|---|---|---|---|---|
| 纯规则启发 | 高 | 中 | 最低(0.5~1d) | 极佳 | MVP / 简单·中等 |
| α-β / expectiminimax | 高 | 高 | 中(2~3d) | 好(需裁剪) | 困难 |
| MCTS | 中 | 高 | 中高(3~4d) | 一般 | 非首选 |
| 规则裁剪 + 浅搜（混合） | 高 | 高 | 中(复用规则) | 好 | 最终形态 |

## 3. 评估函数设计建议

评估函数 `evaluate(state, forSide): number`，约定**正分对 `forSide` 有利**。因为胜利条件非对称，特征分两套，最终 `score = (forSide 视角的己方收益) − (对手视角的己方收益)`，即零和化处理。

### 3.1 终局/近终局项（权重最高，压倒一切）

- 已达成胜利：`±∞`（或 ±100000）。
- `survivorKills`、`zombieKills`：每 1 分都极重要（满 4 即胜）。给大权重，例如每个 survivorKill 对幸存者 +1200、每个 zombieKill 对僵尸 +1200。**临门一脚**（kills==3）再加额外奖励，鼓励/警惕收官。

### 3.2 僵尸方特征

- **每个幸存者的"被包夹进度"**：对每个幸存者数其正交相邻僵尸数 n（0~4）。`n==1` 给中等正分（差一个就感染），`n>=2` 即将/已感染给高分。还要看"补第 2 个所需空格是否存在且僵尸够得着"——可达性折扣。这是僵尸最核心的进攻信号。
- **包夹潜力**：对每个幸存者，统计其正交空邻格中"有己方僵尸在 1~2 步内能到"的数量。
- **控场格数**：场上僵尸数 + 库存（产能）。`zombieReserve` 是战略资源，库存高加分（还能"原地复活"放大感染收益，见 `runInfection:103`）。
- **暴露惩罚**：己方僵尸若正处于"某幸存者相邻且其背后是合法空格"（即可被跳杀）→ 扣分（送分）。
- **聚集度**：僵尸过于分散难形成 2 包夹，轻度奖励僵尸离最近幸存者的曼哈顿距离收敛。

### 3.3 幸存者方特征

- **可达跳杀机会数**：当前 `legalJumps` 总数 + "1 步机动后可形成的跳杀"数（前瞻 1 步），加分。这是幸存者唯一得分手段。
- **跳杀安全性**：跳杀落点若落入"会被 ≥2 僵尸正交包夹"则该跳杀价值打折甚至变负（取分但下回合被感染得不偿失）。
- **与最近僵尸群的安全距离**：每个幸存者的"当前正交相邻僵尸数"，n>=1 强惩罚（一步之遥就可能被补成感染），n==0 安全。
- **贴边惩罚**：幸存者在边/角，机动方向与跳杀方向减少、易被逼死 → 扣分（开局四角是劣势位，应尽快离角）。
- **剩余事件卡期望**：未消耗房子数 × 平均卡价值（ghost 对幸存者偏负——僵尸被传走有时反帮僵尸重定位，但通常中性；shoes/coffee 强正——多走 = 多跳杀机会）。可给"可安全踩到的房子数"一个正权。
- **存活人数**：场上幸存者数，每损失 1 个（= 僵尸 +1 分）极重，已并入 3.1。

### 3.4 可落地加权雏形

伪权重（同一方视角，先调这套再 tune；数值是相对量级，非最终）：

```
W_win                = 100000
W_kill_survivor       = +1200 / survivorKill         (幸存者得分手段)
W_kill_zombie         = +1200 / zombieKill           (僵尸得分手段)
W_near_win_bonus      = +800  当己方 kills==3
僵尸视角:
  W_pincer_n1         = +90   每个 n==1 的幸存者
  W_pincer_n2plus     = +400  每个 n>=2(将被感染)
  W_pincer_potential  = +25   每个"可补的第二格且僵尸够得着"
  W_reserve           = +18   每点库存
  W_zombie_onboard    = +12   每个在场僵尸
  W_zombie_jumpable    = -120  每个可被跳杀的僵尸(送分)
幸存者视角:
  W_jump_now          = +110  每个当前合法跳杀
  W_jump_1ply         = +40   每个 1 步后可形成的跳杀
  W_unsafe_jump_pen   = -90   跳杀落点会被感染
  W_adj_zombie_pen    = -130  每个 n>=1 的己方幸存者(危险)
  W_edge_pen          = -22 边 / -45 角  每个贴边幸存者
  W_house_value       = +30   每个可安全踩到的未消耗房子
```

`evaluate = Σ(己方有利项) − Σ(对手按其视角的有利项)`。叶子节点调用 `runInfection` 的副本来"看清"该状态结算后的真实分，避免静态评估漏掉"这步走完立刻被感染"。建议加 **quiescence**：若叶子状态存在悬而未决的跳杀/感染，再多搜 1~2 ply 直到"静"，消除地平线效应（评估函数最容易在此翻车）。

## 4. 难度分级

通过三个旋钮组合，不需要为每档写不同算法：

| 档 | 算法 | 搜索深度 | 候选 K | ε-随机 | 故意次优 |
|---|---|---|---|---|---|
| 简单 | 纯规则启发 | 0（无搜索） | 全部 | 30% 概率走"合法但非最优"着 | 评分时只看 1 步即时收益，忽略对手反击；遇到送分着不主动避开（轻微） |
| 中等 | 规则裁剪 + α-β | 2~3 ply | 8 | 8% | 评估函数用"基础权重"，不开 quiescence |
| 困难 | 规则裁剪 + expectiminimax | 4~6 ply | 6~8 | 0% | 全权重 + quiescence + 着法排序，正确处理牌堆 chance 节点 |

要点：

- **ε-随机**：以概率 ε 从候选里随机挑一个（而非最优），制造"会犯错"的拟人手感，比"故意走第二好的"更自然，且实现一行。
- **简单档关键**：不是"算得浅"，而是"看得近"——只评估即时收益、不前瞻对手，玩家能用 2 步组合稳定击败它，符合新手期望。
- **困难档关键**：深度 + quiescence + 正确随机建模缺一不可，否则会在"踩房子"和"跳杀地平线"上犯低级错。
- 难度仅改三个参数，封装成 `AIConfig { depth, topK, epsilon, useQuiescence, modelCards }`，UI 出三个按钮即可。

## 5. 集成方案（最小侵入式）

### 5.1 AI 应产出什么

**产出"一串等价于玩家点击的 (r,c) 序列"**，复用 `clickCell` 逐个执行。理由：

- `clickCell` 是唯一玩法入口，复用它 = 规则 100% 一致、0 重复实现、感染/胜负/切边全自动。
- 一个完整的"AI 回合"对应的点击序列举例：
  - 僵尸召唤：`[(r,c)]`（1 次点击，回合自动结束）。
  - 僵尸移动 2 步同子：`[(zr,zc), (t1r,t1c), (zr',zc'?)... ]` —— 先点僵尸进 move 模式，再点目标格，再（同子或另一僵尸）点第二步。
  - 僵尸库存空传送：`[(destR,destC), (zombieR,zombieC)]`。
  - 幸存者移动/跳杀：`[(sr,sc), (destR,destC)]`。
  - 幸存者触发卡：序列后面追加 `pendingCard` 各阶段的点击（鬼魂选僵尸→选落点；靴子/咖啡选幸存者→落点[→落点2]）。AI 需在 `clickCell` 之后检查 `state.pendingCard` 并继续喂点击直到 `pendingCard==null`。

这样 AI 模块对游戏规则零侵入，只是"自动玩家"。

### 5.2 内部搜索需要的纯函数（新增，不改旧代码）

搜索时**不能**用 `clickCell` 试算（它带 UI 语义、改原 state、且 `runInfection` 有副作用）。新增一个独立模块 `src/game/ai.ts`，内部用：

- `cloneState(state)`：`structuredClone`（注意 `housesConsumed` 是 `Set`，`structuredClone` 支持 Set；保险起见手动重建 Set / 数组）。
- `enumerateActions(state, side)`：枚举该方一个完整回合的所有原子动作（召唤×落点 / 传送×(僵尸,落点) / 移动组合 / 幸存者(子,目标) / 含卡分支）。复用 `legalMoves`、`legalJumps`、`emptyCells`。
- `applyAction(cloned, action)`：在克隆态上施加动作 + 调 `runInfection`（复用）+ `checkWinner`（复用）。这是新写的纯函数，但**规则细节直接照搬 `state.ts` 里 `handleZombieClick/handleSurvivorClick` 的逻辑**，保证一致。
- `actionToClicks(action)`：把选定 action 翻译成 `(r,c)` 点击序列，喂给真实 `clickCell`。

> 关键约束：`applyAction` 与 `state.ts` 的真实结算逻辑必须语义等价。建议加一个 dev-only 自检：随机对局里让 `applyAction` 的结果与"`cloneState` + 真 `clickCell` 序列"逐状态比对，发现偏差即报警。这是这套方案唯一的"一致性风险点"，必须用测试守住。

### 5.3 挂钩点（具体到文件 / 函数）

驱动 AI 的最佳位置是 **`src/main.ts`**，不是 `state.ts`。理由：`state.ts` 应保持纯规则、无"谁是 AI"的概念；驱动循环属于应用层。

改动点清单（最小集）：

1. **新增 `src/game/ai.ts`**（唯一新文件）：`chooseAction(state, side, config)` → `Array<{r,c}>`；含 `enumerateActions / applyAction / evaluate / cloneState`。
2. **`src/main.ts`** 顶部：新增模块级变量 `let mode: 'pvp' | 'pve' = 'pvp'`、`let aiSide: Side = 'zombie'`、`let aiConfig: AIConfig`。
3. **`src/main.ts` `render()` 末尾**：调用新函数 `maybeRunAI()`。
4. **`src/main.ts` 新增 `maybeRunAI()`**：
   - 条件：`mode==='pve' && !state.winner && !state.pendingCard && state.turnSide===aiSide`（`pendingCard` 由 AI 自己的点击序列内部消化，不在此触发）。
   - 取 `const clicks = chooseAction(state, aiSide, aiConfig)`。
   - 用 `setTimeout(..., 250~500ms)` 逐个 `clickCell(state, r, c)` 执行（**加节奏感**，否则 AI 回合瞬间结束玩家看不清；每点一步可 `render()` 一次做动画）。
   - 每步后检查 `state.pendingCard`：若 AI 是触发方且仍有 pending，继续喂该 action 预生成的后续点击。
   - 序列执行完，因 `clickCell→runEndOfTurn` 已自动切边，循环自然停（轮到人或对局结束）。
   - 防重入：用一个 `let aiThinking=false` 标志，避免 `render()` 多次触发并发 AI。
5. **`src/main.ts` `shell()`**：在 `top-actions` 加 3 个按钮——「人 vs 人 / 人 vs AI」开关、AI 选边（人当幸存者 or 僵尸）、难度（简单/中等/困难）。改这些设置后走现有 `resetGame()`（保持地图）或 `regenMap()` 重开，并把选择写进上面模块变量 + `aiConfig`。
6. **可选 `src/ui/hud.ts`**：AI 思考时显示"AI 行动中…"提示（读 `aiThinking`），纯展示，可后置。

**不需要改**：`rules.ts`、`types.ts`（除非想把 `AIConfig` 类型放这）、`state.ts`（保持纯规则）、`board.ts`（AI 通过同一 `clickCell` 走，复用现有高亮/动画）。

### 5.4 单人模式开关

- `mode` 切到 `pve` 时，`maybeRunAI()` 在每次 `render()` 后判断"轮到 AI 边"就接管；轮到人时 `onCellClick` 照常工作。两者天然互斥（靠 `turnSide`）。
- AI 选边：`aiSide` 决定 AI 演僵尸还是幸存者，人演另一方。
- 热座 PvP 完全不受影响（`mode==='pvp'` 时 `maybeRunAI()` 直接 return）。

## 6. 推荐方案 + 工作量估算

### 6.1 MVP 推荐：纯规则启发式 AI（第 2.1 节），先做僵尸 AI

**理由**：

1. **零规则侵入、零搜索、即时**——只读 `State` + 调现成纯函数，<1ms 出招，浏览器零卡顿。
2. **风险最低**——不需要写 `applyAction`，绕开"内部结算与 `state.ts` 一致性"这个唯一高风险点（MVP 阶段不引入它）。规则 AI 只做"对每个候选着，clone 一份 state、施加这一着、调 `runInfection` 看结果分、打分、取最高"，clone 深度只需 1 层。
3. **覆盖最大价值**——单人可玩 + 简单/中等两档当场就有，是"能不能单人玩"从 0 到 1 的关键。
4. **它本身就是下一步搜索 AI 的候选生成器和叶子兜底**，不是丢弃型工作。
5. 先做**僵尸 AI**：僵尸方策略更"机械"（包夹是明确的局部模式），规则化收益高、最容易做出"像样对手"；幸存者 AI 涉及跳杀链路规划，稍后用搜索增强更划算。

### 6.2 工作量估算

| 阶段 | 内容 | 估时 |
|---|---|---|
| MVP-1 | `ai.ts`：僵尸规则启发（包夹优先级 + 候选评分 + 1 层 clone 试算）+ `actionToClicks` + `main.ts` 驱动循环 + PvE 开关按钮 + AI 选边/难度 UI | **1.5~2 人天** |
| MVP-2 | 幸存者规则启发（安全跳杀优先 + 避险 + 离角 + 顺路吃卡），凑齐两边都能当 AI | **1 人天** |
| 一致性守护 | dev-only 自检 + 手测 5 局回归（对照 spec §9） | **0.5 人天** |
| 迭代-1（可选） | `applyAction` 纯函数 + α-β（中等档），含与 `state.ts` 等价性测试 | **+2~3 人天** |
| 迭代-2（可选） | expectiminimax 牌堆 chance 节点 + quiescence + 着法排序（困难档）+ 评估函数 tuning | **+2~3 人天** |
| 迭代-3（可选） | 评估权重自博弈调参 / 可选 MCTS 实验 | **+2~4 人天** |

**MVP 总计 ≈ 3~3.5 人天**即可交付"可单人对战 + 三难度（用 ε-随机区分简单/中等，困难留给迭代）"。完整三档强 AI 累计约 7~10 人天。

### 6.3 后续可迭代方向

1. 规则 AI → 规则裁剪 + α-β（第 2.4 混合形态），困难档质变。
2. 引入 expectiminimax 正确处理牌堆，消除"踩房子"决策的盲点。
3. quiescence + 着法排序压制地平线效应（评估函数最大失真来源）。
4. 评估权重自博弈/对人对局数据微调。
5. 若追求极高棋力且评估难调，再实验 rollout 用规则 AI 的"重 playout" MCTS。
6. AI 思考可移到 Web Worker，彻底消除主线程卡顿（当前裁剪后浅搜不必要，深搜阶段再上）。

## 7. 风险与注意事项（落地必读）

1. **`runInfection` 有副作用**：AI 任何"试算"都必须在 `cloneState` 出的副本上调用，绝不能碰真 `state`。
2. **僵尸回合是复合动作**：搜索/枚举的"一步"必须是"一整个僵尸回合的终态"（召唤=1 子操作即结束；移动=2 步组合；传送=2 次点击）。把它当单步会让搜索逻辑错乱。
3. **点击序列的状态依赖**：AI 生成的 `(r,c)` 序列依赖中间态（如先点僵尸进 move 模式，目标格才合法）。`actionToClicks` 必须按 `state.ts` 状态机顺序产出，且驱动循环每点一次后用真实 `state` 校验下一击仍合法（防御 `applyAction` 与真规则的细微偏差）。
4. **`pendingCard` 多阶段**：AI 触发抽卡后，回合"未结束"，驱动循环必须继续喂卡阶段点击直到 `pendingCard==null`，否则卡死在 AI 回合。MVP 简化：AI 触发卡时用固定保守策略消化（如鬼魂传最近威胁僵尸到无害角落 / 靴子咖啡走最大化跳杀或最远离险）。
5. **`applyAction` 一致性是迭代阶段唯一高风险点**：必须有自动对照测试守住，否则 AI"以为"的规则与真规则分叉，行为诡异且难查。MVP 不引入它即可完全规避。
6. **节奏**：AI 回合用 `setTimeout` 分步播放（250~500ms/步），否则瞬间结束破坏可读性与"对战感"。
7. **无外部依赖**：项目纯 Vite+TS、无框架、`devDependencies` 只有 vite/typescript，AI 应保持零依赖纯 TS，契合 spec §1"不做 AI/联网"之外的极简技术栈基调（本次是新增可选模块，不破坏热座 PvP）。

## 附：一句话结论

贴着 `clickCell` 唯一入口做"自动玩家"：**MVP = 纯规则启发式（先僵尸后幸存者）**，零规则侵入、即时、风险最低，约 3~3.5 人天即可单人可玩；后续按"规则裁剪 + 浅层 α-β/expectiminimax + quiescence"路线增量升级到困难档。唯一需要严守的工程纪律是——搜索/迭代阶段引入的 `applyAction` 必须有与 `state.ts` 的等价性自动测试。
